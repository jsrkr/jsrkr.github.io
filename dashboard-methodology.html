Dashboard Notes

Digital Life and Fertility Future

Methodology and interpretation notes for the dashboard, which combines observed state trends with transparent scenario paths so users can compare alternative digital futures without treating them as causal forecasts.

Observed state layer

For each state, the dashboard stores ACS remote-work share, commute inputs, remote-work time saved, fertility measures, population growth, and other direct state-year inputs for 2014-2024. When the local ACS extract does not contain commute-time microdata, the pipeline flags the fallback and uses the best available state, region, or national substitute.

Observed national time-use layer

ATUS provides the benchmark for digital media minutes, broad screen leisure, in-person social minutes, work at home, work away, household work, and unpaid care.

Digital-media and screen-leisure scenarios use these ATUS-based screen-leisure and digital-media time measures. Some broader digital-access measures are not currently used in the public scenario adjustment because they are not consistently populated across states and years.

Reference path

The state fertility projection begins with a reference path anchored to the last observed general fertility rate and extended using the state's observed trend.

Scenario path

Each public scenario path is defined as reference path plus a transparent scenario adjustment. The remote-work scenario uses the change in remote-work time saved relative to the reference path and is anchored to a documented CPS remote-work fertility benchmark; screen scenarios use ATUS-based digital and screen-leisure minutes; care scenarios use ATUS-based household and unpaid-care minutes.

Projection map

The main map shows scenario effect vs reference at the selected horizon, using diverging colors centered at zero.

How to use State and MSA views

State view. The State view is the primary forecast view. State models are based on fewer, larger, more stable geographies and have stronger held-out performance.

Metro patterns view. The Metro patterns view is exploratory. It helps users see whether digital-life scenarios may matter more in some metropolitan labor markets than others, but it is best used for broad spatial patterns rather than exact local forecasts.

Why counties appear in the MSA map. The dashboard estimates MSA-level values but displays them across counties for visibility. Counties inside the same estimated MSA receive the same MSA-level value.

Model choice. Model choice depends on the task. For exact prediction, compare RMSE, MAE, and R2. For map display, also check whether forecasts hit model boundaries and whether the model preserves spatial variation. Tree ML is preferred for MSA display because it preserves metropolitan variation, not because the Metro view is intended for exact local forecasting.

Rankings. MSA rankings are exploratory and should be checked against model diagnostics and scenario assumptions.

MSA county display layer

The Metro patterns view keeps the analytical unit at the identified MSA / CBSA level. County polygons are used only as a fuller display layer: counties inside an estimated MSA inherit the parent MSA value, while micropolitan and outside-CBSA counties remain context only.

For the current MSA scenario build, remote-work and online-matching scenarios may use parent-state proxy inputs merged onto each MSA when fully MSA-native inputs are not available. MSA-specific reference paths are retained, but the scenario shock itself is proxy-based, so those proxy-backed scenarios should not be read as fully local MSA-native estimates.

For the current MSA build, the online-matching scenario applies a common proxy shock across MSAs at each horizon. It should be interpreted as a national overlay rather than MSA-specific online-matching exposure.

Late-horizon MSA reference paths from the statistical baseline and neural-network benchmark often hit the model's lower bound in this build, so Tree ML is preferred for MSA display because it preserves metropolitan variation and avoids lower-bound forecast problems. That preference is about display quality, not about claiming exact metro-level forecasts.

Model reliability labels

State and MSA models are evaluated separately. State forecasts are based on fewer, more stable geographies. MSA forecasts provide finer metropolitan variation, but they are noisier and more vulnerable to lower-bound problems.

The dashboard assigns each model and geography one of four labels: Recommended, Available, Use caution, or Benchmark only. The labels combine held-out validation accuracy, geography coverage, and how often reference or scenario forecasts hit the model's lower or upper bound.

The statistical baseline is the primary State benchmark because it is transparent, stable, and performs well out of sample. Tree ML has strong held-out accuracy and is shown as a robustness benchmark.

When many forecasts hit a bound, maps and rankings can become less informative because place-to-place differences are compressed. In the current bundle, Tree ML is preferred for MSA display because it preserves metropolitan variation better in the forecast horizon. Neural-network results are shown as benchmark or robustness results rather than the main interpretation.

Future dashboard versions may add model-agreement indicators for MSA scenario direction and ranking stability.

Interpretation rule

The central interpretation question is what the marginal digital hour is replacing. A digital hour replacing commuting may have a different fertility implication than a digital hour replacing face-to-face social life.
